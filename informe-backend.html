1. Introducción al proyecto Wiser

Wiser es una plataforma de conexión entre alumnos y profesores particulares. Los alumnos pueden buscar profesores por materia, modalidad, precio y zona geográfica; agendar turnos de clase; comunicarse via chat; y dejar valoraciones. Los profesores gestionan su perfil, disponibilidad horaria, agenda de clases y datos de cobro.

El sistema está compuesto por dos partes: un backend implementado como una API REST en C# con .NET 8, y un frontend móvil desarrollado en React Native con Expo. Ambas partes se comunican exclusivamente a través de HTTP, sin ningún acoplamiento directo de código entre sí.

Este informe documenta exclusivamente la parte del backend: su tecnología, arquitectura, endpoints, seguridad y los momentos en que se comunica con el frontend.

3. Arquitectura general del sistema

El backend sigue el patrón de API REST con una arquitectura de dos capas principales:

  • Capa de presentación (Controllers): recibe las requests HTTP, valida los datos de entrada, ejecuta la lógica de negocio y devuelve respuestas JSON.
  • Capa de acceso a datos (EF Core + AppDbContext): abstrae la comunicación con PostgreSQL mediante el ORM Entity Framework Core, utilizando LINQ para construir queries tipadas.

4.2 Relaciones entre entidades

Las relaciones entre entidades están configuradas en Data/AppDbContext.cs usando la API fluida de Entity Framework Core.

Relación N-N: Profesor ↔ Materia

Un profesor puede enseñar varias materias y una materia puede ser enseñada por varios profesores. EF Core genera automáticamente la tabla intermedia MateriaProfesor con las claves foráneas de ambas entidades.

Relaciones 1-N

  • Profesor → Turnos: un profesor tiene muchos turnos.
  • Alumno → Turnos: un alumno tiene muchos turnos.
  • Profesor → Disponibilidades: un profesor define muchas franjas horarias.
  • Profesor → DatosBancarios: un profesor puede tener sus datos bancarios registrados.
  • Profesor → Valoraciones: un profesor recibe muchas valoraciones.
  • Conversacion → Mensajes: una conversación contiene muchos mensajes.

Relaciones 1-1

  • Usuario → Alumno: cada cuenta de tipo alumno tiene un perfil Alumno asociado.
  • Usuario → Profesor: cada cuenta de tipo profesor tiene un perfil Profesor asociado.
  • Turno → Pago: cada turno tiene exactamente un pago asociado.

Restricciones de integridad

Configuradas en AppDbContext.cs:

  • Restricción única en Conversacion: no pueden existir dos conversaciones entre el mismo alumno y el mismo profesor.
  • Restricción única en Disponibilidad: un profesor no puede tener dos disponibilidades para el mismo día y turno.
  • Check constraint en Valoracion: el campo Puntaje debe estar entre 1 y 5 inclusive.

4.3 Autenticación y manejo de contraseñas

El sistema delega la autenticación a Firebase Authentication (Identity Provider externo de Google). Firebase se encarga de almacenar las contraseñas de forma segura usando su propio sistema de hashing (bcrypt con salt), emite tokens JWT firmados, y gestiona bloqueos por intentos fallidos.

En la base de datos propia del proyecto, la entidad Usuario no almacena contraseñas. En su lugar, contiene el campo FirebaseUid, que es el identificador único que Firebase asigna a cada cuenta. Este UID vincula la identidad de Firebase con el perfil local en PostgreSQL.

Nota: La decisión de delegar la autenticación a Firebase elimina la responsabilidad de gestionar hash+salt propio en la base de datos, trasladando esa responsabilidad a un servicio especializado y auditado.

4.4 Migraciones y seeders

El esquema de la base de datos se gestiona a través del sistema de migraciones de Entity Framework Core. Cada migración es un archivo C# en la carpeta Migrations/ que describe los cambios incrementales aplicados a la base de datos. Para aplicar todas las migraciones y crear las tablas, se ejecuta el comando:

dotnet ef database update

El sistema incluye un seeder de materias implementado en MateriaController.ObtenerTodas(): al recibir el primer request a GET /api/materia, si la tabla está vacía, inserta automáticamente 11 materias predefinidas que cubren los tres niveles educativos (primario, secundario y universitario).

5. Endpoints y lógica del servidor

Corresponde a la Parte 2 de la rúbrica (30 puntos).

5.1 Estructura de controladores y HTTP verbs

El backend expone sus funcionalidades a través de 7 controladores, cada uno agrupando los endpoints relacionados con una entidad o área funcional. Todos los controladores derivan de ControllerBase (sin vistas) y utilizan el atributo [ApiController], que activa automáticamente la validación de modelos.

Controlador Prefijo de ruta Endpoints principales
UsuarioController /api/usuario POST registro, GET por Firebase UID, PUT cuenta
ProfesorController /api/profesor GET búsqueda paginada, GET perfil, PUT perfil, GET/PUT datos bancarios
AlumnoController /api/alumno GET perfil, PUT perfil
TurnoController /api/turno GET disponibles, POST crear, GET por alumno, GET por profesor, PUT estado, PUT cancelar, POST comprobante
ChatController /api/chat GET chats del profesor, GET chats del alumno, GET mensajes (con marca de leído), POST enviar mensaje
ValoracionController /api/valoracion GET valoraciones del profesor, POST crear valoración
MateriaController /api/materia GET todas las materias (con seeder automático)

Los verbos HTTP utilizados siguen la convención REST:

  • GET — consultas de datos (búsqueda de profesores, turnos, chats, mensajes).
  • POST — creación de recursos (turno, mensaje, valoración, comprobante de pago, registro de usuario).
  • PUT — actualización de recursos existentes (perfil del profesor/alumno, estado de turno, cancelación, datos bancarios).

5.2 Validaciones en el servidor

Todos los objetos de transferencia de datos (DTOs) recibidos en los endpoints tienen anotaciones de validación del namespace System.ComponentModel.DataAnnotations. Gracias al atributo [ApiController] en cada controlador, ASP.NET Core valida automáticamente el modelo antes de ejecutar el método del controlador: si el cuerpo del request no cumple las restricciones, devuelve una respuesta 400 Bad Request con detalle de los errores sin necesidad de código adicional.

Las anotaciones utilizadas por DTO son:

5.3 Arquitectura en capas

El backend está estructurado en dos capas:

  1. Capa de Controladores (Controllers/): recibe los requests HTTP, ejecuta la lógica de negocio (verificación de disponibilidad, creación de recursos compuestos) y devuelve respuestas JSON.
  2. Capa de Acceso a Datos (Data/AppDbContext.cs + EF Core): gestiona la comunicación con PostgreSQL a través del ORM, encapsulando las queries en métodos LINQ tipados.
Nota: No se implementó una capa de servicio (Service Layer) ni patrón Repository de forma explícita. La lógica de negocio reside en los controladores, lo que concentra la responsabilidad pero simplifica la estructura del proyecto.

5.4 Manejo de errores y códigos HTTP

Cada endpoint devuelve el código HTTP apropiado según el resultado de la operación:

  • 200 OK — operación exitosa.
  • 400 Bad Request — datos de entrada inválidos (validación automática del modelo).
  • 401 Unauthorized — token JWT ausente o inválido (gestionado por el middleware).
  • 403 Forbidden — el usuario autenticado no tiene permisos para la operación solicitada.
  • 404 Not Found — el recurso solicitado no existe en la base de datos.
  • 409 Conflict — la operación entraría en conflicto con datos existentes (por ejemplo, intentar reservar un turno ya ocupado).
  • 500 Internal Server Error — error no controlado devuelto como JSON por el handler global.

En Program.cs se configura un middleware global de manejo de excepciones (UseExceptionHandler) que captura cualquier excepción no controlada y devuelve una respuesta JSON con el formato {"error": "Error interno del servidor."} en lugar de exponer el stack trace.

5.5 ORM y persistencia con Entity Framework Core

Entity Framework Core 8 es el ORM que abstrae la comunicación con PostgreSQL. Su uso en el proyecto incluye:

  • Include / ThenInclude para cargar relaciones de forma eager loading (evitar N+1 queries).
  • AsQueryable() para construir queries dinámicas con filtros opcionales (búsqueda de profesores).
  • Proyecciones anónimas con Select() para devolver solo los campos necesarios en cada endpoint, sin exponer la entidad completa.
  • AnyAsync(), FirstOrDefaultAsync(), CountAsync() para operaciones de consulta asíncronas.
  • Skip() / Take() para paginación server-side.
  • Enums como strings: todos los enums del dominio (estados de turno, roles, modalidades) se serializan como texto legible usando JsonStringEnumConverter.
  • camelCase automático: configurado en Program.cs con PropertyNamingPolicy = JsonNamingPolicy.CamelCase para que la API devuelva JSON en el formato esperado por el frontend JavaScript.

6. Comunicación backend ↔ frontend

Corresponde a la Parte 3 de la rúbrica (20 puntos).

El frontend (aplicación React Native) y el backend se comunican a través de HTTP REST. No existe ningún acoplamiento de código entre ambas partes: el frontend sólo conoce la URL base de la API y el contrato de los endpoints (rutas, métodos HTTP y formato JSON de los datos).

6.1 Capa de comunicación: apiService.js

Toda la lógica de comunicación con la API está centralizada en el archivo wiser-app/services/apiService.js. Este archivo define una función authFetch() que actúa como wrapper de la función nativa fetch:

const authFetch = async (url, options = {}) => {
  const user = auth.currentUser;          // usuario logueado en Firebase
  const token = user ? await user.getIdToken() : null;
  const headers = {
    ...(options.headers || {}),
    ...(token ? { Authorization: `Bearer ${token}` } : {})
  };
  return fetch(url, { ...options, headers });
};

Esta función obtiene automáticamente el JWT emitido por Firebase para el usuario actualmente logueado y lo agrega a cada request como cabecera Authorization: Bearer <token>. El backend valida ese token en cada request protegido.

Sobre authFetch() se construyen todas las funciones exportadas del servicio, una por cada operación disponible en la API:

  • buscarProfesores(params) — búsqueda paginada de profesores con filtros.
  • obtenerPerfilProfesor(profesorId) — perfil completo de un profesor.
  • actualizarPerfilProfesor(profesorId, data) — edición del perfil.
  • obtenerTurnosAlumno(alumnoId) / obtenerTurnosProfesor(profesorId) — agenda de turnos.
  • crearTurno(data) — reserva de un turno con pago.
  • actualizarEstadoTurno(turnoId, estado) — confirmación o rechazo por el profesor.
  • obtenerChats(profesorId) / obtenerChatsAlumno(alumnoId) — lista de conversaciones.
  • obtenerMensajes(conversacionId, lectorId) — historial de mensajes y marca de leído.
  • enviarMensaje(conversacionId, remitenteId, contenido) — envío de un mensaje.
  • crearValoracion(data) — calificación de un turno realizado.
  • Y más: datos bancarios, comprobante de pago, cancelación de turno, etc.

Momentos de comunicación frontend ↔ backend

6.2 Búsqueda y filtros de profesores

El endpoint GET /api/profesor acepta 5 parámetros de filtro opcionales en el query string:

  • nombre — búsqueda por nombre o apellido (contiene, sin distinción de mayúsculas).
  • materiaId — filtra por ID de materia enseñada.
  • zona — filtra por zona geográfica (texto parcial).
  • precioMaximo — filtra profesores con precio por hora menor o igual al valor indicado.
  • modalidad — filtra por modalidad (presencial, virtual, híbrida).

En el frontend, el alumno dispone de una barra de búsqueda por nombre y chips de selección de materia. Ambos inputs aplican un debounce de 400 ms antes de lanzar el request a la API, evitando requests innecesarios mientras el usuario escribe.

6.3 Paginación

El endpoint de búsqueda de profesores implementa paginación server-side. El backend acepta los parámetros:

  • page — número de página (por defecto: 1).
  • pageSize — cantidad de resultados por página (por defecto: 10, máximo: 50).

La respuesta devuelve un objeto con metadatos de paginación y los resultados ordenados por valoración promedio descendente:

{
  "total": 47,
  "page": 1,
  "pageSize": 10,
  "totalPaginas": 5,
  "datos": [ ... ]
}

El frontend acumula los resultados de páginas sucesivas en el estado local. Muestra un contador "X de Y profesores" y un botón "Ver más resultados" que carga la siguiente página sin reemplazar los resultados anteriores.

6.4 Usabilidad en la comunicación

El frontend implementa varias estrategias para mejorar la experiencia de usuario durante la comunicación con la API:

  • ActivityIndicator: muestra un spinner de carga mientras se espera la respuesta del servidor.
  • Alert.alert(): notifica al usuario en caso de error de red o respuesta de error de la API.
  • RefreshControl: permite al usuario actualizar manualmente la lista de chats y turnos deslizando hacia abajo (pull-to-refresh).
  • useFocusEffect: recarga automáticamente los datos cada vez que el usuario regresa a una pantalla (por ejemplo, al volver a la lista de chats después de leer un mensaje).
  • Polling automático: en la pantalla de chat-detail, los mensajes se recargan cada 5 segundos para mostrar nuevos mensajes sin que el usuario tenga que hacer nada.

7. Master-detail y transacciones

Corresponde a la Parte 4 de la rúbrica (10 puntos).

7.1 Flujos master-detail

La aplicación implementa el patrón master-detail en tres flujos principales:

  1. Profesores: el alumno ve una lista de profesores en su dashboard (master) y puede acceder al perfil completo de cada uno (detail), con información extendida de materias, valoraciones, disponibilidad y botón para agendar.
  2. Chats: la pantalla de chats muestra la lista de conversaciones activas (master). Al tocar una, accede al historial completo de mensajes de esa conversación (detail).
  3. Turnos: la pantalla de mis turnos lista todos los turnos del alumno o el profesor (master). Cada turno muestra su estado de pago, acciones disponibles y la posibilidad de calificar al profesor (detail).

7.2 ABM con cabecera y detalle

El sistema implementa operaciones compuestas donde se crean o modifican simultáneamente una entidad cabecera y sus entidades detalle:

Turno + Pago + Conversacion (POST /api/turno)

Cuando un alumno reserva un turno, el endpoint POST /api/turno crea en una sola operación:

  • El Turno (cabecera) con la información de la clase (fecha, horario, materia, modalidad).
  • El Pago (detalle) asociado al turno, inicializado en estado pendiente con el método de pago elegido.
  • La Conversacion entre el alumno y el profesor, si aún no existía una previa.

Profesor + Disponibilidades + Materias (PUT /api/profesor/{id})

Al editar su perfil, el profesor actualiza en una sola operación:

  • Sus datos básicos (Profesor cabecera): título, descripción, modalidad, precio, zona.
  • Sus Disponibilidades horarias: se eliminan las anteriores y se crean las nuevas.
  • Su relación N-N con Materias: se limpia la colección y se reconstruye con los IDs seleccionados.

7.3 Transacciones con rollback

Las operaciones compuestas que involucran múltiples SaveChangesAsync() están protegidas por transacciones de base de datos explícitas. Esto garantiza que si alguno de los pasos falla, todos los cambios anteriores de esa operación se revierten, evitando datos inconsistentes.

Ejemplo en TurnoController.Crear():

using var tx = await _db.Database.BeginTransactionAsync();
try
{
    _db.Turnos.Add(turno);
    await _db.SaveChangesAsync();          // 1° save: persiste el Turno y genera su Id

    _db.Pagos.Add(new Pago { TurnoId = turno.Id, ... });   // usa el Id del Turno
    if (convExistente == null)
        _db.Conversaciones.Add(new Conversacion { ... });

    await _db.SaveChangesAsync();          // 2° save: persiste Pago y Conversacion
    await tx.CommitAsync();

    return Ok(new { turno.Id, turno.Estado, ... });
}
catch
{
    await tx.RollbackAsync();              // revierte el 1° save si el 2° falla
    return StatusCode(500, "Error al crear el turno.");
}

El mismo patrón se aplica en UsuarioController.Registro(), donde primero se persiste el Usuario para obtener su Id, y luego se crea el Alumno o Profesor asociado.

8. Seguridad

Corresponde a la Parte 5 de la rúbrica (10 puntos).

8.1 Autenticación JWT con Firebase

El flujo de autenticación funciona de la siguiente manera:

  1. El usuario se registra o inicia sesión en el frontend usando Firebase Authentication (email+contraseña).
  2. Firebase emite un token JWT firmado con las claves privadas de Google.
  3. El frontend adjunta ese token en cada request a la API como cabecera Authorization: Bearer <token>.
  4. El backend valida el token en cada request usando el middleware JwtBearer, que descarga automáticamente las claves públicas de Firebase (OIDC discovery) y verifica la firma, el emisor y la audiencia.
  5. Si el token es válido, el request continúa. Si es inválido o está ausente, el middleware devuelve automáticamente 401 Unauthorized.

La configuración en Program.cs especifica:

  • Authority: https://securetoken.google.com/wiser-app-ff1ae (Firebase project ID).
  • Issuer: mismo valor que Authority.
  • Audience: wiser-app-ff1ae (Firebase project ID).
  • ValidateLifetime: true — rechaza tokens expirados.

8.2 Autorización y rutas protegidas

Todos los controladores tienen el atributo [Authorize] a nivel de clase, lo que significa que cualquier endpoint requiere autenticación por defecto. Solo los endpoints que tienen sentido como públicos están marcados explícitamente con [AllowAnonymous]:

Endpoint Acceso Justificación
GET /api/profesor Público Búsqueda de profesores accesible sin cuenta
GET /api/profesor/{id} Público Ver perfil de un profesor sin estar logueado
GET /api/valoracion/profesor/{id} Público Ver reseñas sin cuenta
GET /api/materia Público Necesario en el formulario de registro
POST /api/usuario/registro Público Crear cuenta (primer uso, antes de tener sesión)
GET /api/usuario/firebase/{uid} Público Recuperar perfil al iniciar sesión
Todos los demás endpoints Protegido Requieren JWT válido de Firebase

8.3 Buenas prácticas de seguridad

  • Prevención de SQL Injection: Entity Framework Core parametriza automáticamente todas las queries generadas por LINQ. No se utiliza SQL crudo en ningún punto del backend.
  • CORS: configurado en Program.cs con la política ReactNative, necesaria para que el cliente móvil pueda comunicarse con la API durante el desarrollo.
  • Swagger con Bearer: la interfaz de Swagger incluye un campo de autenticación para ingresar el JWT y probar endpoints protegidos directamente desde la UI.
  • Global exception handler: las excepciones no controladas son capturadas por el middleware global y devueltas como JSON, evitando exponer stack traces al cliente.
  • Brute force: Firebase gestiona internamente los intentos fallidos de login (bloqueos temporales, CAPTCHA), no requiere implementación en el backend propio.

10. Conclusión

El backend de Wiser implementa una API REST completa y funcional sobre .NET 8, PostgreSQL y Entity Framework Core 8. Los puntos fuertes del sistema son:

  • Modelo de datos rico: 11 entidades propias con relaciones 1-N, N-N y 1-1, restricciones de integridad y auditoría completa mediante campos de fecha de creación.
  • Endpoints bien definidos: 7 controladores con más de 20 endpoints que utilizan los verbos HTTP de forma semántica, validaciones en los DTOs y manejo explícito de errores.
  • Seguridad implementada: autenticación JWT con Firebase validada en el servidor, todos los endpoints sensibles protegidos con [Authorize], y prevención automática de SQL Injection vía EF Core.
  • Comunicación robusta con el frontend: la capa apiService.js centraliza todas las llamadas HTTP e incorpora automáticamente el token de autenticación en cada request.
  • Transacciones explícitas: las operaciones compuestas (crear turno + pago + conversación; registrar usuario + perfil) están protegidas con BeginTransactionAsync / RollbackAsync para garantizar consistencia en la base de datos.
  • Paginación y filtros: la búsqueda de profesores soporta 5 filtros opcionales con paginación server-side, con respuesta estructurada que incluye metadatos (total, páginas, datos).

Como área de mejora futura se identifica la incorporación de una capa de servicios (Service Layer) que separe la lógica de negocio de los controladores, y la creación de documentación técnica adicional (diagrama de entidades, guía de instalación) que complemente la documentación de API que ya provee Swagger.
